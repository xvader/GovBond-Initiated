## PROMPT 3 — Security Audit + Hardening

```
You are a smart contract security auditor reviewing the GovBond codebase.
Read every .sol file in contracts/. Then apply all fixes described below.
Do not change any function signatures visible to the frontend unless noted.

─────────────────────────────────────────────────────────────────
AUDIT FINDINGS TO FIX
─────────────────────────────────────────────────────────────────

### HIGH — Reentrancy in GovBondVault
distributeCoupon, redeem, and deposit all make external calls (safeTransfer,
mint, burnFrom) inside loops or without reentrancy protection.

Fix: Import OpenZeppelin ReentrancyGuard. Add nonReentrant modifier to:
  requestDeposit, deposit, requestRedeem, redeem, distributeCoupon, sweepToken.
Pattern: follow checks-effects-interactions — update ALL state (r.claimed = true,
hasPendingDeposit = false, etc.) BEFORE external calls.

Reorder redeem() to:
  1. Load request
  2. Validate (all requires)
  3. Update state: r.claimed = true, hasPendingDeposit = false
  4. External calls: burnFrom, safeTransfer
  5. _trackHolder

### HIGH — Integer overflow risk in maxHolding check (ComplianceModule)
Current: newBalance * 10000 > maxHoldingBps * totalSupply
If newBalance is near type(uint256).max / 10000, this overflows.
Fix: use mulDiv pattern or restructure:
  newBalance > (maxHoldingBps * totalSupply) / 10000
(totalSupply * maxHoldingBps is safe for realistic supply sizes, but document
the assumption that totalSupply < type(uint256).max / 10000 in NatSpec)

### HIGH — No access control on IDRPToken.faucet()
If testnetFaucetEnabled is true, anyone can mint unlimited IDRP.
Fix: Add a daily per-address cooldown:
  mapping(address => uint256) public lastFaucetTime;
  In faucet(): require(block.timestamp >= lastFaucetTime[to] + 24 hours, "Cooldown");
  lastFaucetTime[to] = block.timestamp;

### MEDIUM — GovBondVault.distributeCoupon rounding loss
When (totalCouponPool * bal) / totalSupply has remainder, dust accumulates in vault.
Fix: track totalPaid in the loop. After loop:
  uint256 dust = totalCouponPool - totalPaid;
  if (dust > 0) {
    // Return dust to agent caller
    idrp.safeTransfer(msg.sender, dust);
    emit DustReturned(dust);
  }
Add event DustReturned(uint256 amount).

### MEDIUM — No input validation on BondFactory.deployBond
Add requires:
  require(bytes(name).length > 0 && bytes(name).length <= 64, "Invalid name");
  require(bytes(symbol).length >= 2 && bytes(symbol).length <= 8, "Invalid symbol");
  require(faceValueIDRP >= 100, "Face value too small"); // min Rp 1.00
  require(couponRateBps <= 5000, "Coupon > 50%");
  require(maturityMonths >= 1 && maturityMonths <= 120, "Invalid maturity");
  require(maxSupplyUnits >= 1 && maxSupplyUnits <= 1_000_000, "Invalid supply");

### MEDIUM — GovBondToken forcedTransfer bypasses compliance
forcedTransfer calls _transfer which goes through _update and triggers
compliance. This is intentional but must be documented. However, it also
checks frozen status — a forced transfer on a frozen address will revert,
defeating the regulatory purpose.
Fix: Add a separate internal path. Change forcedTransfer to:
  function forcedTransfer(address from, address to, uint256 amount)
    external onlyRole(AGENT_ROLE) returns (bool)
  {
    // bypass _update hooks for regulatory action
    super._transfer(from, to, amount); // direct ERC20 transfer skipping overrides
    emit ForcedTransfer(from, to, amount);
    return true;
  }
Wait — _transfer in ERC20 calls _update internally. Instead:
  Use a flag: bool private _forcedTransferActive;
  In _update: if (_forcedTransferActive) { super._update(from, to, amount); return; }
  In forcedTransfer: _forcedTransferActive = true; _transfer(from, to, amount); _forcedTransferActive = false;
  Use a try/finally equivalent:
    _forcedTransferActive = true;
    _transfer(from, to, amount);
    _forcedTransferActive = false;
  (Solidity has no finally — document that if _transfer reverts, flag stays true.
  Mitigate: only AGENT_ROLE can call this. Add a resetForcedFlag() admin function
  as emergency reset.)

### LOW — Missing events
Add events where missing:
  - IdentityRegistry: emit events already present — confirm batchRegister emits InvestorRegistered for each
  - GovBondVault.setBondPrice: emit BondPriceUpdated(oldPrice, newPrice)
  - ComplianceModule.blockCountry: emit CountryBlocked(bytes2 country, bool blocked)
  - BondFactory: already has BondDeployed — add ISSUER_ROLE granted event via OZ AccessControl (automatic)

### LOW — IDRP decimals compatibility
IDRP uses 2 decimals. Add a comment in GovBondVault explaining the decimal
math to future auditors:
  // IDRP: 2 decimals. 1 bond unit = faceValueIDRP IDRP base units.
  // Example: faceValue=100_000_000 means Rp 1,000,000.00
  // deposit(100_000_000, ...) → 1 bond unit minted (0-decimal)
  // redeem(1, ...) → 100_000_000 IDRP base units returned

─────────────────────────────────────────────────────────────────
ADDITIONAL HARDENING
─────────────────────────────────────────────────────────────────

1. Add NatSpec (@notice, @param, @return) to all public/external functions
   in all contracts.

2. Add custom errors instead of require strings where gas matters:
   In GovBondVault, replace all require(condition, "string") with
   custom errors: error ZeroAssets(); error PendingDepositExists(); etc.
   Keep string-based requires only in GovBondToken for ERC-3643 compliance
   readability.

3. In IdentityRegistry.batchRegister, add:
   require(investors.length <= 200, "Batch too large"); // prevent gas limit DoS

4. In ComplianceModule, add:
   require(address(identityRegistry) != address(0), "Registry not set");
   in the constructor.

─────────────────────────────────────────────────────────────────
OUTPUT REQUIREMENTS
─────────────────────────────────────────────────────────────────
Write complete final versions of ALL modified contracts:
  contracts/IDRPToken.sol
  contracts/GovBondToken.sol
  contracts/GovBondVault.sol
  contracts/ComplianceModule.sol
  contracts/IdentityRegistry.sol
  contracts/BondFactory.sol

After writing, produce a SECURITY_AUDIT.md file summarizing:
- All findings (HIGH/MEDIUM/LOW/INFO)
- Status (Fixed / Won't Fix + reason)
- Residual risks
- Recommended post-deployment monitoring
```

## Key Design Decisions (for your reference)

**IDRP decimals = 2** (sen), so Rp 1,000,000 = `100_000_000` base units.
**Bond token decimals = 0** (whole units only — no fractional bonds).
**bondPrice** = faceValueIDRP in IDRP base units.
**share math**: `shares = assets / bondPrice` (integer division; revert if remainder).

**EnumerableSet** replaces the external holder array in `distributeCoupon`.
The factory **cannot** grant MINTER_ROLE on IDRPToken automatically — the IDRP admin must do this manually after each bond deploy. The Issuer Portal shows a checklist for this.

**Three frontend files, three audiences**:
- `index.html` → investors (subscribe, hold, redeem)
- `deploy-bond.html` → treasury/issuer (deploy new bonds)
- `admin.html` → operations team (KYC, fulfillment, coupons, compliance)
